# M4 Aggressive status

Status: **in progress; first 400k candidate failed the style gate**.

## What completed

The first residual-adapter candidate completed 400,000 environment steps,
1,444 episodes, and 6,252 PPO updates with zero KL early stops. Its final
checkpoint SHA-256 is
`426a956d58330b1c0edbb117e3a1f0f167034744f10bcafefbce1f79b5d6c12d`.
Training used validation-free league inputs and reports
`test_cases_accessed: false`.

The implementation includes a frozen fair-observation base actor, a
zero-initialized residual style adapter, a copied style policy head,
`D_KL(style || base)`, capped reward components, style-checkpoint inference,
and an append-only paired evaluation path.

## Honest result

A 20-episode validation smoke compared the frozen style-base candidate and the
400k Aggressive candidate across all five script opponents with side swaps.
The ledger was complete and protocol-clean. Overall Skill Retention was 90%,
but neither policy selected an attack action; engagement initiation and valid
attack rates were therefore zero. The Aggressive candidate failed
`engagement_shift`, `valid_attack_rate`, and the per-opponent retention floor.

The raw append-only evidence and its hash-bound manifest are under
`reports/m4/smoke/aggressive-0400000/`. This smoke is a diagnostic result, not
an M4 pass and not publication metric evidence.

## Root cause and proposed correction

The original 100k BC demonstration set contains no learner attack actions.
With the base actor frozen, valid-hit shaping is too sparse to move greedy
inference across the attack-action exploration barrier. Running the full
200-episode evaluation would only repeat an already explicit failure.

The proposed next route is success-filtered Aggressive teacher distillation:
generate public-observation attack opportunities with the existing privileged
teacher, retain only attacks that produce a valid hit, pretrain only the style
adapter and copied policy head, then resume task-reward PPO with KL and the
existing anti-hacking penalties. This proposal requires route approval before
implementation.
